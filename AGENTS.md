# AGENTS.md

本文件用于指导 Codex/子代理在本仓库中持续推进 PhotoView 项目，目标是在上下文变长、任务拆分、记忆压缩后仍能稳定接续，不误删、不跑偏、不重复造轮子。

## 基本原则

- 默认使用简短中文沟通，优先给结论和下一步。
- 开始任何任务前，先读本文件、`docs/plan.md`、`docs/TODO.md`。
- 关键不确定性必须先问用户，不要自行假设后继续执行。
- 不做无关重构，不改无关文件，不清理用户未要求清理的内容。
- 本仓库长期开发任务默认按“功能验收后 commit、阶段完成后 push”执行。
- 不执行 Git reset、checkout、rebase、force push 等可能改写历史或丢失数据的操作，除非用户明确要求。
- 涉及删除、覆盖、迁移数据、部署、外部服务、可能产生费用的操作，必须先确认。
- 代码改动保持小步提交式粒度，每次只完成一个明确任务闭环。

## 技术栈约定

- 桌面框架：Tauri 2。
- 后端：Rust。
- 数据库：SQLite。
- 前端：优先选择轻量、成熟、适合桌面应用的方案；若尚未定型，先在 `docs/TODO.md` 或架构文档中明确后再落地。
- 图片能力：Rust 侧负责扫描、元数据、缩略图、缓存、文件操作；前端负责交互、布局、状态展示。
- 跨平台目标：Windows 10/11 优先，同时预留 macOS/Linux。

## 项目目标

PhotoView 是一个本地图片查看器与合集管理工具，最终目标是完成 `docs/plan.md` 中列出的全部功能。阶段划分只用于控制实现顺序和上下文规模，不代表裁剪范围。

核心能力包括：

- 导入文件夹为合集。
- 扫描图片并提取元数据。
- 生成和缓存缩略图。
- 浏览合集与图片网格。
- 打开图片查看器，支持翻页、缩放、旋转、全屏。
- 管理合集、标签、收藏、最近浏览。
- 后续支持搜索、重复检测、文件监听、备份恢复和跨平台打包。

## 推荐执行顺序

1. 先完成 `docs/TODO.md` 阶段 0，锁定全量功能边界、架构边界和验收标准。
2. 再完成阶段 1，搭建 Tauri、SQLite、扫描和基础 API。
3. 再完成阶段 2，形成可导入、可浏览、可查看图片的基础闭环。
4. 核心体验稳定后推进阶段 3 的管理能力。
5. 最后推进阶段 4 的高级功能与阶段 5 的质量、性能、打包。

## 完成标准

任何 TODO 被标记完成前，必须满足：

- 功能已按当前阶段范围实现，不把核心逻辑留成空壳或 TODO 注释。
- 已处理错误状态、加载状态、空状态和基础边界条件。
- 已完成必要验证，并在最终回复或 `PROGRESS.md` 记录验证方式。
- 涉及数据或文件操作时，已确认不会破坏用户真实数据。
- 相关文档、TODO 或进度记录已同步更新。

每完成一个可验收功能后，必须：

- 完成对应验证。
- 更新 `docs/TODO.md` 和 `PROGRESS.md`。
- 检查 `git status`，只暂存本功能相关文件。
- 创建一次语义清晰的 Git commit。
- 最终回复说明 commit hash。

阶段完成前，必须满足：

- 当前阶段所有关键 TODO 已完成或明确移到后续阶段。
- `pnpm`/Rust/Tauri 的可用检查已通过；如果项目还未初始化，说明原因。
- 没有悬挂的开发服务器、测试浏览器或后台任务。
- `PROGRESS.md` 已更新到可续接状态。

每完成一个阶段后，必须：

- 确保该阶段最后一个功能已 commit。
- 再次运行阶段级验证。
- 将当前分支 push 到远端跟踪分支。
- 最终回复说明远端分支和最新 commit hash。

## 子代理拆分规则

当任务较大时，按职责拆给子代理或分阶段处理：

- 架构 subagent：Tauri/Rust/SQLite/API 边界、目录结构、缓存策略。
- 产品 subagent：全量功能验收标准、实现顺序、功能依赖关系。
- Tauri 工程 subagent：项目初始化、窗口、菜单、权限、系统能力。
- 数据库 subagent：schema、migration、CRUD、索引。
- 文件扫描 subagent：格式识别、目录导入、增量扫描、元数据。
- 图片解码 subagent：格式支持、大图加载、异常图片处理。
- 缩略图 subagent：生成队列、磁盘缓存、清理策略。
- 前端 UI subagent：布局、页面、虚拟滚动、交互状态。
- 查看器 subagent：缩放、翻页、旋转、全屏、快捷键。
- 管理功能 subagent：合集、图片、标签、收藏、最近浏览。
- 搜索 subagent：全局搜索、高级筛选、结果跳转。
- QA/性能 subagent：测试、性能基准、跨平台冒烟。

子代理产出必须包含：

- 做了什么。
- 改了哪些文件。
- 如何验证。
- 剩余风险或下一步。

子代理冲突控制：

- 同一时间不要让多个子代理修改同一文件或同一模块。
- 分派前必须说明每个子代理负责的目录、文件或功能边界。
- 子代理完成后先汇总，再由主代理统一合并和验证。
- 如果发现其他代理或用户改动了同一文件，必须重新读取后再继续。

## 上下文控制

为了避免上下文过大，每次任务必须控制读取范围：

- 优先读 `AGENTS.md`、`docs/TODO.md`、当前相关文件。
- 如果存在 `PROGRESS.md`，必须一并读取。
- 搜索文件优先使用 `rg` 或 `rg --files`。
- 不整段粘贴大文件，必要时只读相关行。
- 不把长日志、长依赖输出、完整构建输出塞进回复。
- 如果发现任务涉及多个模块，先写 3-6 行小计划，再执行。
- 每完成一个小阶段，必须更新 `docs/TODO.md` 或 `PROGRESS.md`，方便压缩后续接。

## 记忆压缩与续接

如果上下文被压缩或会话中断，恢复后必须先执行：

1. 读取 `AGENTS.md`。
2. 读取 `docs/TODO.md`。
3. 读取 `PROGRESS.md`。
4. 查看当前文件状态和最近改动。
5. 根据未完成 TODO 和当前进度继续，不从头重做。

必须维护一个轻量进度文件：

- 文件名：`PROGRESS.md`。
- 只记录当前阶段、已完成事项、正在做的事项、阻塞点、下一步。
- 不记录长篇解释。
- 每次完成较大任务后更新一次；长任务中断前也要更新。

推荐格式：

```md
# Progress

## Current

- 阶段：
- 正在做：
- 下一步：

## Done

- 

## Blocked

- 
```

## 文件修改规则

- 手动编辑文件优先使用 `apply_patch`。
- 不使用脚本批量改写项目文件，除非任务确实需要。
- 新增依赖前先确认是否已有同类依赖。
- 新增依赖必须说明用途，并优先选择维护活跃、跨平台可靠的依赖。
- 依赖版本、Tauri 插件、前端框架和包管理器一旦确定，必须写入项目文档或 `PROGRESS.md`。
- 修改配置文件前先理解现有配置。
- 不删除用户文件、图片、数据库、缓存目录，除非用户明确要求。
- 对可能破坏数据的功能，必须默认走确认弹窗、回收站或软删除。

## 验证规则

每个代码任务结束前至少做一种验证：

- 格式化或静态检查。
- 单元测试。
- 构建检查。
- 手动运行关键命令。
- 前端/Tauri 交互验收。

关键阶段推进前必须尽量通过：

- 前端 lint/typecheck/build。
- Rust fmt/clippy/test。
- Tauri dev 或 build 的基础检查。
- 核心用户流程的手动或自动化验收。

如果无法验证，最终回复必须说明原因和风险。

使用 Playwright 或浏览器验收后，结束前必须关闭测试页面或浏览器，避免悬挂会话。

## UI/交互规则

- 这是工具型桌面应用，界面应安静、清晰、高效。
- 首屏应是可用的管理界面，不做营销落地页。
- 优先支持键盘快捷键、右键菜单、拖拽、多选、排序、筛选。
- 大列表和图片网格必须考虑虚拟滚动。
- 图片加载、缩略图生成、导入扫描必须有进度和错误状态。
- 删除、移动、覆盖等危险操作必须明确确认。

## 性能边界

- 万级图片合集必须能流畅滚动。
- 缩略图生成应走后台队列，避免阻塞 UI。
- 图片打开优先使用缓存和渐进式状态。
- 导入目标：1000 张图片小于 30 秒。
- 已缓存图片打开目标：小于 200ms。
- 大图浏览必须控制内存，避免一次性加载过多原图。

## 数据与安全

- 数据库必须有 migration。
- migration 必须可重复执行，失败时不能破坏已有数据。
- 需要数据迁移时，必须先备份或提供回滚策略。
- 文件路径要跨平台处理，不硬编码 Windows 路径分隔符。
- 所有文件操作必须处理权限、文件不存在、重名、跨磁盘移动失败。
- 删除图片默认优先移到回收站；直接删除必须二次确认。
- 数据库备份/恢复、重建索引、批量删除必须先有明确确认流程。
- 开发和测试默认使用 fixture 或临时目录，禁止直接对用户真实图片库做批量删除、移动、迁移测试。
- 数据库和缩略图缓存必须设计一致性策略，包括缓存失效、孤儿缓存清理、DB lock 处理。

## Tauri 安全边界

- 文件系统访问必须限制 scope，只允许用户选择或应用数据目录内的路径。
- Tauri command 必须使用明确 allowlist，不暴露任意文件读写或 shell 执行能力。
- IPC 参数必须校验路径、ID、分页、排序字段和操作类型。
- 前端必须配置合理 CSP，不加载不可信远程脚本。
- SVG 和异常图片按不可信输入处理，避免脚本执行、路径穿越和解码崩溃。
- 打开外部链接、文件位置或系统应用前，必须由用户触发。

## 后台任务规则

- 导入、缩略图生成、重复检测、文件监听必须走后台任务，不阻塞 UI。
- 长任务必须支持进度、取消、失败重试和错误汇总。
- 应用崩溃或退出后，任务状态必须能恢复或安全重建。
- 文件监听必须防抖，避免重复扫描和数据库写入风暴。
- 并发任务必须限制数量，避免 CPU、内存、磁盘 IO 被打满。

## 跨平台风险

- Windows 必须考虑长路径、文件锁、盘符、UNC 路径、回收站差异。
- macOS/Linux 必须考虑大小写敏感、权限、符号链接、隐藏文件。
- 所有平台必须正确处理 Unicode 文件名、特殊字符、同名文件和路径规范化。
- 目录扫描必须明确是否跟随符号链接，默认不跟随，避免循环扫描。
- 文件排序、创建时间、修改时间在不同平台含义不同，UI 文案要避免误导。

## 回复要求

- 默认 3-6 行中文总结。
- 说明已完成内容、验证结果、下一步。
- 不展开长背景，除非用户要求。
- 遇到错误时只说明原因、影响、下一步。
